AI Trust Receipt

A candidate governance specification for verifiable, auditable records of consequential AI actions.

The Problem

AI agents increasingly perform or initiate consequential actions on behalf of people—changing access, modifying data, delegating authority, or affecting third parties. Existing records are often operational, fragmented, inaccessible to affected people, or insufficient to reconstruct what occurred, whether valid authorization existed, what evidence informed the action, and how the result can be contested. Financial and commercial systems normalize transaction receipts. AI-mediated actions that materially affect a person’s status, eligibility, access, or resources still lack a broadly adopted equivalent.

Structural Gap
This candidate proposes an interoperable, user-legible receipt format that couples a consequential AI action to its authority, evidence, outcome, and remedy.

What the AI Trust Receipt Is

The AI Trust Receipt is a candidate governance mechanism that couples a consequential AI action to a durable, machine-verifiable event record. The release includes a normative candidate technical specification, an executable reference implementation, machine-readable schemas, a practitioner workbook, and a conformance assessment framework. It is designed to be tested, challenged, independently reviewed, and adapted through documented implementation pathways.

Five Core Functions

Pre-Execution Gate Authority, evidence, and remedy capability must exist before any consequential action executes.
Canonical Event Record One structured object from which the machine receipt, human disclosure, and integrity verification are derived.
Human-Readable Disclosure Rendered from the same canonical record, preventing an independently authored narrative from contradicting the machine record.
R1–R12 Conformance Twelve noncompensatory requirements. Failure of any single requirement cannot be offset by strength in the other eleven.
Remedy Reference A contestation pathway must be operational before execution, rather than supplied as a static policy page after the action.

Claim Boundary

This is a candidate governance and implementation package. It is not an adopted standard, certification program, legal-compliance determination, external audit, or regulatory approval. A receipt records what an accountable system represents as having occurred and the evidence by which that representation can be checked. Trustworthiness depends on truthful inputs, sound governance, operational controls, and independent scrutiny where warranted. The included SHA-256 method detects a content mismatch only relative to a trusted expected digest or trusted receipt channel; it does not authenticate an issuer or define a production digital-signature suite.
